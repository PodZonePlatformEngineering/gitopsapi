# ApplicationDeployment — Ownership Model and Promotion Workflow

This document covers two closely related topics for `ApplicationDeployment` records:

1. **HTTPRoute ownership** — who generates and manages the routing manifest, and what that implies
2. **Chart version promotion** — how `pipeline_stage` links a deployment record to the Change Pipeline
3. **API-level approval gates** — which roles can perform which operations at each stage

---

## 1. HTTPRoute Ownership

An `ApplicationDeployment` can declare external hostnames via `external_hosts`. Whether an
HTTPRoute manifest is generated from that list depends on the `http_route` field:

| `http_route` | `external_hosts` | HTTPRoute manifest | Owner |
|---|---|---|---|
| set | non-empty | generated by GitOpsAPI at `gitops/gitops-apps/<app>/<app>-httproute-<cluster>.yaml` | **GitOpsAPI-owned** |
| null | non-empty | not generated | **Externally-owned** |
| null | empty | not generated | — (no routing) |

### GitOpsAPI-Owned HTTPRoutes

When `http_route` is set, GitOpsAPI generates a `gateway.networking.k8s.io/v1 HTTPRoute`
manifest in the same PR as the Kustomization entry. The lifecycle of this manifest is coupled
to the ApplicationDeployment:

- **Create**: HTTPRoute written alongside the Kustomization entry (single PR).
- **Update** (`PATCH`): If `http_route` or `external_hosts` changes, a new PR replaces the
  HTTPRoute manifest.
- **Delete**: When the deployment record is deleted, the HTTPRoute manifest is removed in the
  same PR as the Kustomization removal.

Each cluster gets its own independent `http_route` spec. Promoting an app from dev to prod
does **not** copy the HTTPRoute spec — the operator must supply `http_route` on the production
`ApplicationDeployment` explicitly.

**When to use**: 3rd-party or external apps where GitOpsAPI controls all routing config —
keycloak, forgejo, minio, etc. The simplest path: declare the hosts and the Gateway target,
and the API generates the HTTPRoute.

### Externally-Owned HTTPRoutes

When `http_route` is null but `external_hosts` is non-empty, the operator is declaring routing
intent without handing manifest generation to GitOpsAPI. The external hosts list is still
written as an annotation (`gitopsapi.podzone.net/external-hosts`) on the Kustomization entry,
where it can be read by:

- **Cluster provisioning tooling** — cert-manager Certificate SANs and Gateway listener
  hostnames are derived from `ClusterSpec.external_hosts` at provisioning time.
- **Networking operators** — teams managing HTTPRoutes outside GitOpsAPI can read the
  annotation to discover intended host assignments.

**When to use**: Internally-built applications where routing is managed as part of the
application's own GitOps manifests (the app repo contains its own HTTPRoute); or where routing
is controlled by a dedicated networking layer outside GitOpsAPI's scope.

### Ownership Implications at Promotion Time

When a chart version is promoted from dev → ete → prod via the Change Pipeline, only the
`chart_version_override` field is updated on the target cluster's `ApplicationDeployment`.
Routing configuration (`external_hosts`, `http_route`) is **not carried forward** automatically:

- Each cluster's routing is declared independently on that cluster's deployment record.
- A dev deployment may have no `external_hosts` (internal access only), while the prod
  deployment has `external_hosts` + `http_route` for public-facing routing.
- This independence is intentional — dev and prod routing topology frequently differ.

## 2. Chart Version Promotion Workflow

### Role of `pipeline_stage`

The `pipeline_stage` field on an `ApplicationDeployment` declares the cluster's role in
the promotion pipeline for this application:

| Value | Meaning |
|---|---|
| `"dev"` | Development cluster — first target; unrestricted promotion cadence |
| `"ete"` | End-to-end test cluster — receives promoted versions from dev |
| `"production"` | Production cluster — final target; highest approval bar |
| `null` | No pipeline participation (3rd-party/external apps, or manual management) |

When `pipeline_stage` is `null`, the `ApplicationDeployment` is self-contained — chart version
changes are made directly via `PATCH /api/v1/application-deployments/{id}` without going
through a pipeline promote call.

### Promotion Flow

A `PipelineSpec` names the three cluster IDs and the application being promoted:

```
POST /api/v1/pipelines                      → create pipeline instance for a specific version
POST /api/v1/pipelines/{name}/promote       → promote to "dev" | "ete" | "production"
```

When `promote` is called, the service:

1. Looks up the `ApplicationDeployment` record for `(app_id, target_cluster_id)`.
2. Writes a PATCH to the target cluster's `{cluster}-apps` repo, updating
   `chart_version_override` on the Kustomization's values override file.
3. Opens a PR in the target cluster's `{cluster}-apps` repo with the appropriate stage label.
4. Returns the PR URL in the `PipelineResponse`.

Flux CD reconciles the merged PR, which triggers a HelmRelease upgrade on the target cluster.

### Stage Sequence and Gates

```
[build_manager] POST /pipelines/{name}/promote  target_stage=dev
        │
        ▼
   dev cluster  ──── dev testing ────►  [build_manager] approve PR
        │
        ▼ POST /pipelines/{name}/promote  target_stage=ete
   ete cluster  ──── E2E testing ───►  [build_manager] approve PR
        │                               ETE tests must pass
        ▼ POST /pipelines/{name}/promote  target_stage=production
  prod cluster  ──── serving ─────────  [build_manager + cluster_operator] approve PR
```

No stage may be skipped: promoting to `production` without an ETE deployment in the pipeline
history is rejected at the service layer.

### HTTPRoute Behaviour During Promotion

The promote operation updates `chart_version_override` only. It does **not** modify:
- `http_route` — routing must be configured per-cluster on each `ApplicationDeployment`
- `external_hosts` — independently declared per cluster
- `values_override` — cluster-specific values are not carried forward during promotion

If an app requires different routing config on each cluster, configure each cluster's
`ApplicationDeployment` independently before or after promotion.

## 3. API-Level Approval Gates

### Create / Delete

| Endpoint | Method | Allowed Roles | Notes |
|---|---|---|---|
| `/api/v1/application-deployments` | POST | `cluster_operator`, `security_admin` | Creates Kustomization entry (and HTTPRoute if `http_route` set) |
| `/api/v1/application-deployments/{id}` | DELETE | `cluster_operator`, `security_admin` | Removes Kustomization (and HTTPRoute if GitOpsAPI-owned) |

### Update

| Endpoint | Method | Allowed Roles | Notes |
|---|---|---|---|
| `/api/v1/application-deployments/{id}` | PATCH | `cluster_operator`, `build_manager`, `security_admin` | `build_manager` may update `chart_version_override`; `cluster_operator` may update routing or values |

### Pipeline Promotion

| Endpoint | Method | Allowed Roles | Notes |
|---|---|---|---|
| `POST /api/v1/pipelines` | POST | `build_manager` | Create a pipeline instance for a version |
| `POST /api/v1/pipelines/{name}/promote` | POST | `build_manager` | Promote to any stage |

### PR Approval and Merge

PRs opened by promotion carry a `stage:` label that drives required-approver enforcement:

| Stage label | Required approvers (API layer) | Required approvers (git forge branch protection) |
|---|---|---|
| `stage:dev` | `build_manager` | `build_manager` |
| `stage:ete` | `build_manager` | `build_manager` |
| `stage:production` | `build_manager` + `cluster_operator` | `build_manager` + `cluster_operator` |

Approval is enforced at two levels:

1. **GitOpsAPI** — `POST /api/v1/prs/{pr_number}/approve` checks the caller's role against
   the PR's stage label before forwarding the approval to the git forge.
2. **Git forge branch protection** — the target branch (main on each cluster repo) requires
   the same approvers; a PR cannot be merged without satisfying both.

### Role Reference

| Role | Keycloak group | Key permissions |
|---|---|---|
| `cluster_operator` | `cluster-operators` | Provision/delete clusters; create/delete application deployments; approve production PRs |
| `build_manager` | `build-managers` | Create pipelines; promote versions; update chart overrides; approve dev/ete/prod PRs |
| `security_admin` | `security-admins` | Create/delete application deployments; manage credential objects |
| `senior_developer` | `senior-developers` | Read-only access to pipelines, history, deployments |

## Summary

| Concern | Decision |
|---|---|
| HTTPRoute generation | Opt-in per deployment via `http_route` field; null = external ownership |
| Routing at promotion | Not carried forward — configure per-cluster independently |
| Promotion flow | dev → ete → prod via `POST /pipelines/{name}/promote`; `build_manager` required; no stage skipping |
| Dev gate | `build_manager` approves PR |
| ETE gate | `build_manager` approves PR; ETE tests must pass |
| Production gate | `build_manager` + `cluster_operator` must both approve PR |
| `pipeline_stage: null` | No pipeline — chart version managed directly via `PATCH` |
