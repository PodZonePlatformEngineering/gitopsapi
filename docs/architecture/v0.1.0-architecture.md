## 1. System Overview

### 1.1 Purpose

GitOpsAPI is a REST API that implements GitOps-based management of Kubernetes clusters and applications. All mutations flow through the git repository as the single source of truth: the API writes manifests to feature branches, opens pull requests for human review and approval, and relies on Flux CD to reconcile the approved state onto target clusters.

### 1.2 High-Level Architecture

```text
┌──────────────────────────────────────────────────────────────────┐
```

### 1.3 Data Flow Summary

**Write path** (any mutation):

```text
Client → API → validate → GitService (feature branch + write files + commit + push)
       → GitHubService (open PR with labels)
       → Human approves PR in GitHub
       → GitHubService (merge PR, tag commit)
       → Flux CD reconciles on target cluster
```

**Read path** (interrogation):

```text
Client → API → GitService (read YAML from local clone)
              OR K8sService (query cluster Kubernetes API)
       → API → response
```

### 1.4 External Dependencies

| Dependency | Purpose | Protocol | Auth |
| --------- | ------- | -------- | ---- |
| GitHub API | PR creation, merge, tagging, repo create/archive | HTTPS REST | `GITHUB_TOKEN` (PAT) |
| GitOps repo (management) | Object store (clusters, pipelines) | HTTPS | `GITHUB_TOKEN` injected into clone URL by `_auth_url()` |
| Per-cluster repos (`{cluster}-infra`, `{cluster}-apps`) | Application Kustomization wiring + HelmRelease storage | HTTPS | `GITHUB_TOKEN` injected into clone URL by `_auth_url()` |
| Management cluster K8s API | Kubeconfig extraction, CAPI queries | HTTPS | `MGMT_KUBECONFIG_SECRET` |
| Target cluster K8s APIs | Status, logs, undeploy checks | HTTPS | Per-cluster kubeconfig |
| Keycloak (via OAuth2 Proxy) | Authentication + role derivation | HTTP headers | Injected by proxy |
| SOPS / age | Secret encryption at rest | Local process | `SOPS_AGE_KEY_SECRET` |
| Flux CD (on clusters) | GitOps reconciliation | Watches git repo | Flux deploy key (per repo) |

## 2. Component Architecture

### 2.1 API Layer — FastAPI Routers

All routers mounted at `/api/v1`. Role enforcement via `require_role()` dependency on every endpoint.

| Router | File | Endpoints | Min Role |
| ------ | ---- | --------- | -------- |
| Clusters | `routers/clusters.py` | `GET/POST /clusters`, `GET/PATCH /clusters/{name}`, `POST /clusters/{name}/suspend`, `DELETE /clusters/{name}` | `cluster_operator` |
| Applications | `routers/applications.py` | `GET/POST /applications`, `GET/PATCH /applications/{name}`, `DELETE /applications/{name}/disable`, `POST /applications/{name}/enable` | `build_manager` |
| Application Configs | `routers/application_configs.py` | `GET/POST /application-configs`, `GET/PATCH/DELETE /application-configs/{id}` | `build_manager` |
| Pipelines | `routers/pipelines.py` | `GET/POST /pipelines`, `GET /pipelines/{name}`, `POST /pipelines/{name}/promote`, `GET /pipelines/{name}/history`, `GET /pipelines/{name}/changes`, `POST /pipelines/{name}/changes` | `build_manager` |
| PRs | `routers/prs.py` | `GET /prs`, `GET /prs/{number}`, `POST /prs/{number}/approve`, `POST /prs/{number}/merge` | `build_manager` |
| Repositories | `routers/repositories.py` | `GET /repositories` (cluster repo listing) | `cluster_operator` |
| Status | `routers/status.py` | `GET /status`, `GET /status/{cluster}`, `GET /status/{cluster}/resources`, `GET /status/{cluster}/resources/{kind}/{namespace}/{name}`, `GET /status/{cluster}/resources/{kind}/{namespace}/{name}/logs` | `senior_developer` |

**Startup sequence** (`lifespan` context manager):

```python
async with lifespan(app):
    git = GitService()
    await git.init()   # clone or pull gitops repo
```

The git clone is the only startup I/O. If it fails the pod does not become ready (readiness probe fails).

**React frontend serving**: In production containers, `frontend/dist/` is built by the Node stage and mounted at `/`. FastAPI serves it as `StaticFiles` only if the directory exists — so the same image works with or without the built frontend.

### 2.2 Auth Layer

**File**: `api/auth.py`

Authentication is fully delegated to the OAuth2 Proxy sidecar + Keycloak. The API never validates tokens directly.

```text
Request → OAuth2 Proxy → decodes JWT → injects headers:
  X-Forwarded-User: martin
  X-Auth-Request-Groups: cluster-operators,build-managers
```

Role mapping (first matching group wins):

| Keycloak Group | API Role |
| ------------- | -------- |
| `cluster-operators` | `cluster_operator` |
| `build-managers` | `build_manager` |
| `senior-developers` | `senior_developer` |

**`require_role(*roles)`** is a factory that returns `Depends(_RoleChecker(*roles))`. `_RoleChecker` is a **callable class** (not a closure) whose `__call__` has its own `Depends(_extract_caller)` nested inside. FastAPI's dependency resolver handles nested `Depends` correctly only with the callable class pattern.

Usage:

```python
caller: CallerInfo = require_role("build_manager")          # correct — callable class pattern
# NOT: caller = Depends(require_role("build_manager"))      # WRONG — double-wraps Depends
```

Also maps the `security-admins` group to the `security_admin` role (added after v0.1.0):

| Keycloak Group | API Role |
| -------------- | -------- |
| `cluster-operators` | `cluster_operator` |
| `build-managers` | `build_manager` |
| `senior-developers` | `senior_developer` |
| `security-admins` | `security_admin` |

**Local dev fallback**: `GITOPSGUI_DEV_ROLE` env var bypasses group header requirement. The value must be a role string (e.g. `cluster_operator`), not the Keycloak group name. Never set in production.

**Error responses**:

- `401 Unauthorized` — no recognised role in headers (unauthenticated or unknown group)
- `403 Forbidden` — authenticated but wrong role for the operation

### 2.3 Service Layer

Services are instantiated per-request (stateless construction; shared `GitService._repo` class-level singleton).

#### GitService (`services/git_service.py`)

Single responsibility: git repository I/O. Uses **GitPython** with **HTTPS + PAT auth** (SSH removed in v0.1.6).

Key design decisions:

- **Instance-based (not singleton)**: each `GitService(repo_url, local_path)` instance wraps one git repo. Multiple instances exist for the management repo and per-cluster repos. `repo_url=None` reads `GITOPS_REPO_URL`; `repo_url=""` raises on first use.
- **HTTPS + PAT auth**: `_auth_url()` injects `GITHUB_TOKEN` into the clone URL (`https://<TOKEN>@github.com/...`). No SSH keys, no ssh-agent, no mounted volumes for keys.
- **`asyncio.to_thread()`**: all blocking git I/O is offloaded to a thread pool to avoid blocking the async event loop.
- **Feature branch discipline**: every write goes via `create_branch()` → `write_file()` → `commit()` → `push()`. No direct commits to `main`.
- **Dev flags**: `GITOPS_SKIP_INIT=1` skips the clone (uses existing local repo); `GITOPS_SKIP_PUSH=1` skips `git push`; `GITOPS_SKIP_GITHUB=1` uses `LocalPRStore` instead of GitHub API.

Branch naming convention:

| Operation | Branch Pattern |
| --------- | ------------- |
| Cluster provision | `cluster/provision-{name}-{uuid8}` |
| Cluster update | `cluster/update-{name}-{uuid8}` |
| App add | `application/add-{name}-{uuid8}` |
| App disable | `application/disable-{name}-{uuid8}` |
| App enable | `application/enable-{name}-{uuid8}` |
| Pipeline create | `pipeline/create-{name}-{uuid8}` |
| Pipeline change | `pipeline/change-{name}-{changeId}-{uuid6}` |
| Pipeline promote | `pipeline/promote-{name}-{stage}-{uuid6}` |

#### GitHubService (`services/github_service.py`)

Single responsibility: GitHub PR lifecycle. Uses **PyGitHub** for real GitHub, or **LocalPRStore** for local development.

**LocalPRStore**: file-backed JSON PR state at `{GITOPS_LOCAL_PATH}/.local-prs.json`. Enables full local end-to-end testing without GitHub access. Supports real git merges via GitPython squash merge.

**PR label convention**:

- Resource type: `cluster` | `application` | `pipeline` | `promotion`
- Stage: `stage:dev` | `stage:ete` | `stage:production`

**Required approvers by stage** (embedded in `_STAGE_REQUIRED_ROLES`):

| Stage | Required Roles |
| ----- | ------------- |
| `dev` | `build_manager` |
| `ete` | `build_manager` |
| `production` | `build_manager`, `cluster_operator` |

**Deployment tagging**: after PR merge, `tag_deployment(commit_sha, tag)` applies a lightweight git tag for deployment provenance audit trail.

#### ClusterService (`services/cluster_service.py`)

Manages cluster CAPI provisioning manifests in the GitOps repo.

**GitOps repo paths**:

```text
gitops/cluster-charts/{name}/{name}-values.yaml    ← cluster-chart Helm values
gitops/cluster-charts/{name}/{name}.yaml           ← Namespace + HelmRepository + HelmRelease
gitops/cluster-charts/{name}/kustomization.yaml    ← Kustomize config (ConfigMap generator)
gitops/cluster-charts/{name}/kustomizeconfig.yaml  ← nameReference config
```

Reads derive cluster list from directory enumeration (`list_dir`), not from a database.

#### AppService (`services/app_service.py`)

Manages application HelmRelease manifests and cluster assignment. In the multi-repo layout, app manifests live in the cluster's `-apps` repo and Kustomization wiring lives in the cluster's `-infra` repo. Routing helpers from `repo_router.py` are used to target the correct repo per cluster.

**Multi-repo paths**:

```text
{cluster}-apps repo:
  gitops/gitops-apps/{name}/{name}.yaml              ← HelmRepository + HelmRelease
  gitops/gitops-apps/{name}/{name}-values.yaml       ← Default chart values
  gitops/gitops-apps/{name}/{name}-values-{cluster}.yaml  ← Per-cluster values override
  gitops/gitops-apps/{name}/kustomization.yaml
  gitops/gitops-apps/{name}/kustomizeconfig.yaml

{cluster}-infra repo:
  clusters/{cluster}/{cluster}-apps.yaml             ← Flux Kustomization entry (enable/disable)
```

**Disable/Enable mechanism**: The `clusters/{cluster}/{cluster}-apps.yaml` file in the `-infra` repo contains multiple YAML documents. `_comment_app_block()` and `_uncomment_app_block()` use regex to selectively comment/uncomment only the target app's `Kustomization` document, leaving all other apps unaffected.

**Known gap (CC-055)**: `list_applications()` and `get_application(name)` return empty results in multi-repo production because there is no cluster registry to enumerate all clusters.

#### PipelineService (`services/pipeline_service.py`)

Manages change pipelines and deployment history.

**GitOps repo paths**:

```text
pipelines/{name}/pipeline.yaml                          ← pipeline spec + promotion state
pipelines/{name}/changes/{changeId}/change.yaml         ← application change record
pipelines/{name}/history/{releaseId}/deployment.yaml    ← deployment record
pipelines/{name}/history/{releaseId}/tests/results.yaml ← test results
```

#### RepoRouter (`services/repo_router.py`)

Factory module added in v0.1.5 to support the multi-repo-per-cluster layout. Provides four factory functions that construct correctly-targeted `GitService` / `GitHubService` instances for a given cluster:

```python
git_for_apps(cluster)     # GitService → https://github.com/{ORG}/{cluster}-apps.git
git_for_infra(cluster)    # GitService → https://github.com/{ORG}/{cluster}-infra.git
github_for_apps(cluster)  # GitHubService → {ORG}/{cluster}-apps
github_for_infra(cluster) # GitHubService → {ORG}/{cluster}-infra
```

The `GITHUB_ORG` env var determines the org prefix. `_owner()` falls back to splitting `GITHUB_REPO` if `GITHUB_ORG` is unset.

Services that use the management repo set `self._git = GitService()` / `self._gh = GitHubService()` in their constructor. Services that need per-cluster routing call the `repo_router` helpers inline and inject mocks in tests via `patch("gitopsgui.services.X.git_for_infra", ...)`.

#### K8sService (`services/k8s_service.py`)

Queries live Kubernetes clusters. In v0.1.0, most methods are **stubs** (`raise NotImplementedError`) pending full implementation.

**Implemented**: `get_undeploy_status()` — checks namespace phase (gone/active/terminating/unknown) and surfaces finalizer-blocked resources when terminating.

**Not yet implemented**: `list_all_flux_status()`, `get_cluster_flux_status()`, `list_resources()`, `describe_resource()`, `get_logs()`.

### 2.4 Data Layer — Git as Object Store

The GitOps repository is the database. There is no relational DB, no cache, no message queue.

**Reads**: `GitService.read_file()` reads from the local clone (`/tmp/gitops-repo`). The clone is refreshed on startup and on `checkout_main()` (post-merge). Reads between refreshes see the state at last clone/pull — eventual consistency by design.

**Writes**: All writes create a feature branch, write YAML files, commit, push, and open a PR. The write is not reflected in the cluster until the PR is merged and Flux reconciles (minutes to hours depending on Flux interval).

## 3. Deployment Architecture

### 3.1 Kubernetes Resources

Deployed via Helm chart `charts/gitopsapi/` to namespace `gitopsapi`.

```text
Namespace: gitopsapi
├── Deployment/gitopsapi           ← 1 replica, port 8000
│   ├── initContainer: (none)
│   ├── container: gitopsapi
│   │   ├── env (from values):
│   │   │   ├── GITOPS_REPO_URL
│   │   │   ├── GITOPS_BRANCH
│   │   │   ├── GITHUB_REPO
│   │   │   ├── GITHUB_ORG
│   │   │   └── GITOPS_LOCAL_PATH
│   │   ├── env (from secrets):
│   │   │   ├── GITHUB_TOKEN        ← secret: gitopsapi-github-token / key: token
│   │   │   ├── MGMT_KUBECONFIG_SECRET ← secret: gitopsapi-mgmt-kubeconfig / key: kubeconfig
│   │   │   └── SOPS_AGE_KEY_SECRET ← secret: gitopsapi-age-key / key: key.txt
│   │   ├── volumeMounts:
│   │   │   └── /tmp/gitops-repo    ← emptyDir (git clone workspace)
│   │   └── probes:
│   │       ├── liveness: GET /health  (delay 15s, period 30s)
│   │       └── readiness: GET /ready  (delay 10s, period 10s)
│   └── volumes:
│       └── gitops-repo ← emptyDir
│   Note: SSH key volume removed in v0.1.6. All git auth is HTTPS+PAT via GITHUB_TOKEN.
├── Service/gitopsapi              ← ClusterIP, port 80 → 8000
└── HTTPRoute/gitopsapi            ← Gateway API route
    ├── parentRef: gateway (kube-system)
    ├── hostname: gitopsgui.podzone.cloud
    └── backendRef: gitopsapi:80
```

### 3.2 Secret Pre-requisites

The following Kubernetes Secrets must exist in the `gitopsapi` namespace before deployment:

| Secret Name | Key | Contents |
| ---------- | --- | -------- |
| `gitopsapi-github-token` | `token` | GitHub Personal Access Token (used for HTTPS git auth and GitHub API) |
| `gitopsapi-mgmt-kubeconfig` | `kubeconfig` | Management cluster kubeconfig (base64 or plain) |
| `gitopsapi-age-key` | `key.txt` | SOPS age private key |

`gitopsapi-ssh-key` was removed in v0.1.6. All git transport is now HTTPS + PAT; no SSH key secrets or volumes are required.

### 3.3 Container Build (Multi-stage Dockerfile)

```text
Stage 1: node:20-alpine (frontend-build)
  - npm ci
  - npm run build → dist/

Stage 2: python:3.11-slim (api)
  - pip install -e . (from pyproject.toml)
  - COPY src/ (Python sources)
  - COPY --from=frontend-build dist/ → src/gitopsgui/frontend/dist/
  - CMD: uvicorn gitopsgui.api.main:app --host 0.0.0.0 --port 8000
```

The single image contains both the React frontend (static files) and the Python API. On build infrastructure (erectus, 192.168.1.201), only `Dockerfile.api-only` is used — no Node/frontend stage.

### 3.4 Helm Chart

The gitopsapi Helm chart is published to GitHub Pages at `https://podzoneplatformengineering.github.io/gitopsapi`. Current versions:

- **Chart version**: 0.1.3
- **App version**: v0.1.6

The `docs/index.yaml` is the Helm repo index. The HelmRepository CR on the cluster points to `https://podzoneplatformengineering.github.io/gitopsapi`. To publish a new chart version:

```bash
helm package charts/gitopsapi/
cp gitopsapi-0.1.X.tgz docs/
helm repo index docs/ --url https://podzoneplatformengineering.github.io/gitopsapi
git add docs/ gitopsapi-0.1.X.tgz
git commit -m "chore: publish chart 0.1.X"
git push
```

### 3.5 External Network Access

```text
Internet / Platform team
    ↓
freyr:8081  (SSH port forward to openclaw Gateway)
    ↓
openclaw Gateway (192.168.4.179:80)
    ↓  [Host: gitopsgui.podzone.cloud required]

```

## 7. Technology Selection

### 7.1 Python — Language Choice

**Chosen for**: GitOps ecosystem library availability, rapid development, strong typing via Pydantic, excellent async support.

**Alternatives considered**:

- **Go**: Native Kubernetes ecosystem language; better raw performance. Rejected because Python has superior library support for the specific integration points (GitPython, PyGitHub, python-sops, kubernetes-client). The operational profile is I/O-bound (git, GitHub API, K8s API) — Python async is adequate.
- **TypeScript/Node.js**: Good async primitives and HTTP ecosystem. Rejected because GitOps tooling libraries (git, Kubernetes) are less mature in the Node ecosystem.

**Performance mitigation**: All blocking I/O uses `asyncio.to_thread()`, keeping the event loop free. The API is not compute-bound; P95 latency is dominated by GitHub API round-trips (~500ms) and git operations (~1-2s for clone/push).

### 7.2 FastAPI

**Chosen for**:

- **Automatic OpenAPI docs** at `/docs` — essential for a team API with multiple consumers
- **Pydantic integration** — request validation and response serialisation with zero boilerplate
- **Async-first** — `async def` handlers with `await` throughout; `asyncio.to_thread()` for blocking I/O
- **Dependency injection** — `Depends()` pattern for auth (`require_role`) is clean and testable

**Alternatives considered**:

- **Flask**: Synchronous by default; no automatic schema generation. Would require Flask-RESTX or similar for API docs.
- **Django REST Framework**: Too heavy; ORM and admin not needed; config overhead high for this use case.

### 7.3 GitPython

**Chosen for**: Pythonic API over git operations; handles clone, branch, commit, push without subprocess management; HTTPS + PAT auth via URL injection (`_auth_url()`) is simple and requires no key management infrastructure.

**Alternative**: `subprocess` + `git` CLI. Rejected because error handling and parsing of git output is fragile; GitPython provides structured exceptions and repo state.

**Note**: SSH support was present in earlier versions but removed in v0.1.6. All git transport is now HTTPS + `GITHUB_TOKEN` injected into the clone URL.

### 7.4 PyGitHub

**Chosen for**: Official GitHub REST API client; covers PRs, reviews, labels, tags; auto-handles pagination and rate limiting.

**Alternative**: `httpx` / raw HTTP to GitHub API. Rejected because PyGitHub provides typed objects and handles authentication, retry, and pagination automatically.

### 7.5 SOPS + age Encryption

**Chosen for**:

- **Flux native support**: Flux CD has built-in SOPS decryption (no sidecar needed on clusters)
- **age key simplicity**: Single key file; no GPG key server required
- **Git-safe**: Encrypted YAML looks like YAML; diff-friendly; version controlled

**Alternatives considered**:

- **Sealed Secrets**: Requires controller on every cluster; public key per cluster; no cross-cluster secret sharing
- **HashiCorp Vault**: Adds significant infrastructure complexity; another system to manage
- **External Secrets Operator**: Requires external secret backend (Vault, AWS SM, etc.)

### 7.6 Helm (Application Packaging)

**Chosen for**: Industry standard; values-based parameterisation enables per-environment overrides; versioned charts support promotion workflow; OCI and HTTP chart repositories.

**Alternative**: Kustomize-only. Kustomize has no native chart versioning or chart repository concept — would require manual copying of YAML per environment. Helm's `sourceRef` + `version` fields make chart promotion a single field change.

### 7.7 Kubernetes Gateway API (HTTPRoute)

**Chosen for**: Replaces Ingress; supports hostname-based routing with less annotation noise; Cilium implements it natively without an additional ingress controller.

**Alternative**: Kubernetes Ingress. Rejected in favour of Gateway API as the platform standard (Cilium-native, no NGINX/Traefik needed).

## 8. Failure Modes Analysis

### 8.1 GitHub API Unavailable / Rate Limited

**Scenario**: GitHub API returns 5xx, rate limit exceeded, or network unreachable from pod.

**Impact**: All write operations fail at the `create_pr()` step. Feature branch may already be pushed to GitHub (partial state).

**Current handling**: Exception propagates to FastAPI exception handler → 500 Internal Server Error. Branch remains on GitHub without a PR.

**Mitigation (v0.1.0)**:

- Kubernetes Secret for `GITHUB_TOKEN` — token is correct and valid at deploy time
- `GITOPS_SKIP_GITHUB=1` for local dev (no GitHub dependency)

**Recommended improvements**:

- Retry with exponential backoff on 5xx and rate limit responses
- Return 503 with `Retry-After` header to client
- Health check validates GitHub token validity on startup

### 8.2 GitOps Repo Clone Failure on Startup

**Scenario**: `GITHUB_TOKEN` missing/invalid, repo URL wrong, network unreachable at pod start.

**Impact**: `GitService.init()` raises exception in `lifespan()`. FastAPI startup fails. Pod enters CrashLoopBackOff. Readiness probe (`GET /ready`) never passes.

**Current handling**: Exception bubbles up; pod restarts.

**Mitigation (current)**:

- `GITHUB_TOKEN` injected from `gitopsapi-github-token` Secret — correct PAT must be pre-created
- `GITOPS_SKIP_INIT=1` for local dev

**Recommended improvements**:

- Validate `GITHUB_TOKEN` and `GITOPS_REPO_URL` at startup before attempting clone
- Surface specific error (repo not found vs auth failure vs network) in pod logs

### 8.3 Concurrent Write Race Condition

**Scenario**: Two simultaneous requests both call `create_branch()`, which pulls `main` and creates a new branch. Both may create branches from the same HEAD. PR merge is sequential but branch creation is not mutex-protected.

**Impact**: Low risk in v0.1.0 (single replica, sequential git operations via `asyncio.to_thread()`). Higher risk if scaled to multiple replicas.

**Current handling**: No explicit locking. `asyncio.to_thread()` serialises within a single process via the GIL on git operations, but this is not guaranteed.

**Recommended improvements**:

- Add asyncio `Lock` around branch creation + push sequence
- For multi-replica, use a distributed lock (Redis, Kubernetes lease)

### 8.4 Missing Kubernetes Secrets

**Scenario**: One of the three required secrets (`gitopsapi-github-token`, `gitopsapi-mgmt-kubeconfig`, `gitopsapi-age-key`) is absent at deploy time.

**Impact**: Pod starts (secrets are env vars, not volume mounts for these three), but first API operation requiring the missing value returns 500.

**Current handling**: Missing env var → empty string → downstream failure (GitHub auth fails, K8s client fails, SOPS fails). Error message may be confusing.

**Recommended improvements**:

- Validate all required env vars at startup (`lifespan()`); log clear error and refuse to start if any are missing

### 8.5 GitOps Repo Manifest Corruption

**Scenario**: API writes invalid YAML to a feature branch (e.g., template rendering bug). PR is merged. Flux attempts to reconcile invalid manifests.

**Impact**: Flux Kustomization enters `False/NotReady` state. No further reconciliation until fixed. Other Kustomizations on the same cluster are unaffected (Flux processes them independently).

**Current handling**: No pre-commit YAML validation. Bad manifests can reach main.

**Recommended improvements**:

- Validate generated YAML with `yaml.safe_load()` before committing
- Run `helm template` dry-run before pushing HelmRelease manifests
- Flux `spec.suspend: true` as an emergency stop

### 8.6 Flux Reconciliation Delay

**Scenario**: PR merged successfully but application does not update on cluster for 30 minutes (Flux interval).

**Impact**: User sees successful PR merge but stale application version. Not a failure — expected behaviour.

**Current handling**: By design. Flux polls at configured intervals (`interval: 30m` for HelmRelease).

**Mitigation**:

- Use `flux reconcile helmrelease {name}` to force immediate reconciliation (requires kubectl access)
- Reduce Flux interval for dev cluster (e.g., `interval: 5m`)
- GitOpsAPI future enhancement: call Flux API to trigger immediate reconcile after PR merge

### 8.7 Target Cluster Unreachable (Status Queries)

**Scenario**: Target cluster K8s API is down or kubeconfig is stale when status endpoint is called.

**Impact**: Status endpoints return 504 or exception. Other API operations (write path via git) are unaffected.

**Current handling** (v0.1.0): `get_undeploy_status` returns `namespace_phase: "unknown"` on `config.load_kube_config()` exception. Other status endpoints are stubs.

**Recommended improvements**:

- Timeout on all K8s API calls (10s default)
- Return degraded status (partial data) rather than failing the entire response
- Per-cluster health state in `AggregateStatus`

## 9. Monitoring Strategy

### 9.1 Health Check Endpoints

Dedicated health endpoints are implemented (fixed in v0.1.5+):

- `GET /health` — liveness probe: returns `{"status": "ok"}` immediately, no dependencies checked.
- `GET /ready` — readiness probe: returns 503 until the management repo git clone completes in `lifespan()`.

The Helm chart probes target these endpoints (not `/api/v1/status`). The v0.1.0 bug (probes hitting the stub status endpoint) is resolved.

**Enhanced readiness** (recommended future work):

- Check `GITHUB_TOKEN` is non-empty
- Check `GITOPS_REPO_URL` is non-empty
- Validate GitHub token has correct scopes on startup

### 9.2 Structured Logging

FastAPI uses Python `logging`. Recommended JSON log format:

```json
{
  "timestamp": "2026-03-11T20:00:00Z",
  "level": "INFO",
  "message": "Application created: gitopsapi",
  "request_id": "abc123",
  "user": "martin",
  "application_name": "gitopsapi",
  "pr_url": "https://github.com/..."
}
```

Log levels:

| Level | Events |
| ----- | ------ |
| `INFO` | Cluster/app/pipeline created, PR opened, PR merged |
| `WARNING` | GitHub rate limit approaching, transient K8s error (retry) |
| `ERROR` | Missing secret, invalid spec, unrecoverable service error |
