# gitopsapi — Zero-Trust Bootstrap Design

**Status:** Design / Specification (further additions expected)
**Author:** Martin Colley
**Captured:** 2026-03-28

---

## Motivation

gitopsapi acts as the control plane for a GitOps cluster fleet. Its own startup must
not depend on pre-shared credentials, external identity providers, or human-held
secrets injected at deploy time. The "first key" is the cluster admin's existing
Kubernetes access — everything else is derived from that.

---

## Trust Hierarchy

```
k8s admin access (cluster owner — holds kubecontext to gitopsapi host cluster)
  │
  └─► bootstrap admin password  (generated by gitopsapi, retrieved via kubectl)
        │
        └─► operator provides git repo + credentials via API/GUI
              │
              └─► gitopsapi generates SOPS keypair
                    │  private key → K8s Secret in gitopsapi namespace
                    │  public key  → committed to infra repo (.sops.yaml)
                    │
                    └─► branch protection + PR guardrails (role-enforced)
                          │
                          └─► any permitted actor can add secrets
                                (encrypt with public key, commit, PR)
                                only Flux/k8s can decrypt (holds private key)
```

---

## Bootstrap Flow

### Phase 1 — First Boot

1. gitopsapi pod starts. No bootstrap secret found in its own namespace.
2. gitopsapi generates a random admin password and writes it as a K8s Secret:
   ```
   Secret: gitopsapi-bootstrap-token
   Namespace: gitopsapi
   Data:
     password: <generated>
     expires_at: <now + 24h>    # short-lived — must be consumed
   ```
3. gitopsapi reads `ConfigMap: gitopsapi-config` in its namespace (if present) for
   any pre-staged configuration (git provider, org, notification webhook).
4. gitopsapi logs:
   ```
   Bootstrap token written. Retrieve with:
     kubectl get secret gitopsapi-bootstrap-token -n gitopsapi \
       -o jsonpath='{.data.password}' | base64 -d
   ```
5. All API endpoints return 503 (Service Unavailable) except `/healthz` and
   `/auth/login`. The system is not operational until bootstrap is complete.

### Phase 2 — Operator Retrieves Credentials

The cluster owner — who already holds a kubecontext with access to the gitopsapi
namespace — runs the kubectl command above and retrieves the generated password.

**Trust anchor:** possession of k8s namespace access is the proof of ownership.
No out-of-band credential is required.

### Phase 3 — Operator Provides Git Repository

Via the API or GUI, the operator:

1. POSTs a git repository URL and access credentials (PAT or deploy key with
   `contents: write` + `administration: write` scope for branch protection).
2. Optionally provides a `self_cluster` block:
   ```yaml
   name: management
   tunnel_id: <cloudflare-tunnel-uuid>
   ingress_connector:
     enabled: true
     namespace: cloudflared
   ```
   This self-registers the host cluster in the gitopsapi registry, closing the
   management-cluster gap (see analysis: CC-079 post-mortem).

### Phase 4 — gitopsapi Bootstraps the Repository

gitopsapi performs all of the following atomically (one commit, one PR, or direct
push to a bootstrap branch depending on configured workflow):

1. **Generates SOPS age keypair:**
   - Private key → K8s Secret `sops-age` in gitopsapi namespace
   - Public key → committed to repo at `.sops.yaml`

2. **Scaffolds repo structure** (if repo is empty):
   ```
   clusters/<host-cluster>/
     flux-system/
     infrastructure.yaml
   gitops/
     gitops-<host-cluster>/
       00-prerequisites/
       01-infrastructure/
   sops-keys/
   .sops.yaml                ← age public key
   ```

3. **Self-registers host cluster:**
   If `self_cluster` was provided, writes the cluster entry to
   `clusters/<host-cluster>/clusters.yaml` SOPS-encrypted with the new public key.

4. **Sets up branch protection:**
   - `main` branch: require PR, require at least 1 approval, no direct push
   - Optional: require specific reviewers per role (configurable via API)

5. **Commits, pushes, notifies:**
   - Pushes to repo
   - Emits K8s Event: `gitopsapi bootstrap complete`
   - Deletes the bootstrap token secret (it is no longer needed)
   - API becomes fully operational

### Phase 5 — Operator Bootstraps Admin Password Rotation

On first login with the bootstrap password, the API forces a password change.
The bootstrap token secret is deleted from the namespace at this point if not
already gone.

### Phase 6 — Operator Adds Workflow Actors

The operator adds users to the gitopsapi instance via the API, assigning roles:

| Role | Permissions |
|---|---|
| `cluster_operator` | Full cluster CRUD, SOPS bootstrap, gateway wiring |
| `deployer` | Application deploy, pipeline promote |
| `reviewer` | PR approval (GitHub collaborator), read-only API |
| `observer` | Read-only API, status queries |

Optionally, gitopsapi provisions the corresponding GitHub collaborators on the
infra repo (requires admin-scoped token provided in Phase 3).

## Ongoing Secret Addition (Any Actor)

Once bootstrapped, **any actor in a permitted role** can add secrets to the
repository without access to the private SOPS key:

```bash
# Actor has the repo checked out; .sops.yaml contains the public key
sops --encrypt --in-place mysecret.yaml
git add mysecret.yaml
git push → PR → role-appropriate reviewer approves → merge
```

Flux decrypts on the cluster using the private key in the `sops-age` K8s Secret.
The actor never sees or touches the private key. The encrypted ciphertext is safe
to commit.

**Implication for CC-079 and similar:** the `sops-write-tunnel-token` helper and
secretctl vault patterns are transitional. Post-bootstrap, any token/password/secret
is added via this standard encrypted-PR flow.

## Key Rotation

### Planned Rotation

```
POST /clusters/{name}/rotate-sops-key
```

gitopsapi:
1. Generates a new SOPS age keypair
2. Decrypts every encrypted file in the repo with the old key
3. Re-encrypts every file with the new key
4. Updates `.sops.yaml` with the new public key
5. Replaces `sops-age` K8s Secret with the new private key
6. Commits and pushes (directly to main, no PR — operator-only action)
7. Emits K8s Event: `sops key rotated`

**Note:** Role-players do not need to take any action for planned rotation —
gitopsapi re-encrypts all secrets on their behalf.

### Unplanned Rotation / Complete Restart

In the event of a full cluster loss (private key gone, K8s state gone):

1. Operator re-runs Phase 1–4 (re-bootstrap):
   - Retrieves new bootstrap password via kubectl
   - Provides git repo URL again (repo and its content survive)
   - gitopsapi generates a **new** SOPS keypair
   - Pushes new `.sops.yaml` public key to repo
2. gitopsapi **cannot re-encrypt existing secrets** — it no longer holds the old
   private key. The old encrypted secrets in the repo are unreadable.
3. gitopsapi notifies all actors:
   ```
   K8s Event: sops key rotated — actor re-encryption required
   ```
4. **Each role-player re-adds their secrets:**
   ```bash
   # Actor decrypts with their own copy of the secret (they hold the plaintext)
   sops --encrypt --age <new-public-key> --in-place mysecret.yaml
   git push → PR → approve → merge
   ```

**Key design principle:** Actors hold the plaintext. The cluster holds the private
key. Worst case, the cluster is rebuilt and actors re-encrypt — no secrets are
permanently lost provided actors retain their plaintext copies.

## Backup Implications

| Asset | Where it lives | Backup owner | Recovery path |
|---|---|---|---|
| SOPS private key | K8s Secret `sops-age` | gitopsapi / cluster backup | Re-bootstrap if lost; actors re-encrypt |
| SOPS public key | `.sops.yaml` in repo | Git history | Always recoverable from repo |
| Encrypted secrets | Git repo | Git hosting (GitHub) | Decryptable only if private key intact |
| Plaintext secrets | Actor-held | Each actor | Re-encrypt with new key on recovery |
| Admin password | Actor-held (after rotation) | Each actor | Re-bootstrap flow if lost |
| Git credentials | Actor-held / gitopsapi namespace | Operator | Re-provide via API on re-bootstrap |
| Cluster kubeconfigs | CAPI secrets on management cluster | Cluster backup | Regenerated by CAPI on cluster recovery |

**Recommendation:** Back up the K8s `sops-age` secret as part of standard cluster
backup (Velero or equivalent). If the backup is intact, re-bootstrap is not needed —
restore the secret and restart gitopsapi.

## Repository Taxonomy

Two distinct categories of repo are in scope. This distinction matters for understanding
which SOPS key applies, who the actors are, and what secrets are in scope.

### 1. gitopsapi Instance Repo

The repo bootstrapped in Phase 4. It holds gitopsapi's own operational context:

- gitopsapi schema and cluster registry (`clusters/<host-cluster>/clusters.yaml`)
- gitopsapi's own Flux manifests (GitRepository, Kustomizations, HelmRelease)
- gitopsapi's own SOPS keypair reference (`.sops.yaml`, private key in `sops-age` secret)
- Secrets that gitopsapi itself needs to operate: git credentials, Cloudflare API tokens,
  notification webhook URLs

**Actors:** gitopsapi operator and cluster_operators. These are the people who configure
and run gitopsapi itself.

**SOPS key:** gitopsapi's own age keypair, generated at bootstrap.

### 2. Managed Cluster Repos

Repos that gitopsapi provisions and manages on behalf of workload clusters
(e.g. `gitopsdev-infra`, `gitopsdev-apps`, `management-infra`). Each managed cluster has:

- Its own SOPS age keypair, generated by `POST /clusters/{name}/sops-bootstrap`
- Its own `.sops.yaml` referencing its public key
- Infrastructure and application manifests specific to that cluster
- Secrets in scope for that cluster's workload (database passwords, API tokens, etc.)

**Actors:** The role-players for that cluster (deployers, reviewers, etc.) — not
necessarily the same people as the gitopsapi operator.

**SOPS key:** Per-cluster age keypair. Private key stored in that cluster's `sops-age`
K8s Secret, and also encrypted under the gitopsapi instance key in the instance repo's
`sops-keys/` directory (for cross-cluster recovery access).

### Key distinction

The gitopsapi instance repo is the root of trust. The instance SOPS key can decrypt the
per-cluster private keys (stored encrypted in the instance repo). This means:

- gitopsapi instance recovery → all per-cluster keys recoverable → no actor re-encryption needed
- Per-cluster recovery only → only that cluster's actors need to re-encrypt
- Full loss (instance + all clusters) → actors re-encrypt against new keys, starting with instance repo

## Actor Re-Encryption Scripts

gitopsapi includes interactive shell scripts in the instance repo (and optionally in
each managed cluster repo) to guide actors through re-encryption after a key rotation
or cluster recovery.

### Script location

```
scripts/
  sops-reencrypt.sh        ← instance repo re-encryption
  secrets-manifest.yaml    ← machine-readable list of expected secrets (see below)
```

Each managed cluster repo gets its own copy scaffolded by gitopsapi at cluster
bootstrap time.

### Script behaviour

```
$ ./scripts/sops-reencrypt.sh

gitopsapi secret re-encryption helper
SOPS public key: age1q33m5a5hezkgh...  (read from .sops.yaml)

Secrets requiring re-encryption:
  1. gitops/gitops-management/02-network/cloudflare-tunnel-token.sops.yaml
  2. sops-keys/gitopsdev.agekey.enc
  3. sops-keys/gitopsete.agekey.enc
  [ exit ]

Select secret to re-encrypt (or 'x' to exit): 1

Secret: cloudflare-tunnel-token
Description: Cloudflare tunnel token for management cluster (podzone-management)
Enter value: ****

Encrypting... done.
Written: gitops/gitops-management/02-network/cloudflare-tunnel-token.sops.yaml

Secrets requiring re-encryption:
  2. sops-keys/gitopsdev.agekey.enc
  3. sops-keys/gitopsete.agekey.enc
  [ exit ]

Select secret to re-encrypt (or 'x' to exit): x

Reminder: push the repo when all secrets are complete.
  git add -p && git commit -m "chore: re-encrypt secrets after key rotation" && git push
```

### secrets-manifest.yaml

gitopsapi maintains a machine-readable manifest of every encrypted file in the repo,
used by the script to build the menu. It is updated whenever a secret is added or
removed via the API, GUI, or PR:

```yaml
secrets:
  - path: gitops/gitops-management/02-network/cloudflare-tunnel-token.sops.yaml
    name: cloudflare-tunnel-token
    description: Cloudflare tunnel token for management cluster (podzone-management)
    owner_role: cluster_operator
    fields:
      - name: token
        prompt: "Cloudflare tunnel token"
  - path: sops-keys/gitopsdev.agekey.enc
    name: gitopsdev-age-key
    description: SOPS age private key for gitopsdev cluster
    owner_role: cluster_operator
    fields:
      - name: key
        prompt: "Age private key (AGE-SECRET-KEY-...)"
```

### Notification on key rotation / cluster recovery

When gitopsapi detects that re-encryption is required (either after `POST /rotate-sops-key`
or after re-bootstrap), it:

1. Emits a K8s Event in its namespace: `sops-reencryption-required`
2. Commits an updated `scripts/secrets-manifest.yaml` to the repo marking all secrets
   as `status: needs-reencryption`
3. Notifies the operator via the configured notification channel (webhook, K8s Event,
   or GUI banner) with the message:

   ```text
   SOPS key has been rotated. Actors must re-encrypt their secrets.
   Run scripts/sops-reencrypt.sh in the repo, then push.
   ```
